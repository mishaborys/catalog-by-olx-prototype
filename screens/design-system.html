Buttons

Button hierarchy from highest to lowest emphasis. Each level has a clear role — avoid mixing levels for the same weight of action on one screen. Buttons marked Baseline are confirmed from auth screens. Buttons marked Proposed are suggested additions not yet used everywhere.

Primary Baseline
<button class="btn-dark-olx">
  Register
</button>

<!-- Disabled state -->
<button class="btn-dark-olx" disabled>
  Disabled
</button>
Use for: main form submissions (Register, Login, Reset Password, Save). One per screen. Full-width in form contexts — add style="max-width:…" to constrain in wider layouts.
Teal — Secondary Positive Proposed
<button class="btn-teal">
  Start Import
</button>

<!-- Disabled state -->
<button class="btn-teal" disabled>
  Disabled
</button>
Use for: constructive secondary actions (Start Import, Confirm, Sync). Auto-width. Proposal: standardise padding to match btn-dark-olx (.65rem vs current .65rem — already consistent).
Outlined — Neutral Secondary Baseline
<button class="btn-outline-olx">
  Resend Email
</button>
Use for: neutral secondary actions paired with a primary button (e.g. Resend, Edit, Export). Currently full-width — consider auto-width variant for inline use.
Ghost — De-emphasised / Back Baseline
<button class="btn-ghost"
  onclick="showScreen(2)">
  Back to sign in
</button>
Use for: Cancel, Back, Skip — always placed below the primary action. Never use as the only button on screen.
Danger — Destructive Proposed
<button class="btn-danger">
  Delete Account
</button>
Use for: irreversible destructive actions only (Delete Account, Remove Integration). CSS class btn-danger is defined and ready to use — not yet placed on any screen.
Small Outlined — Inline / Table Actions
<button class="btn-sm-outline">
  Edit
</button>
Use for: table row actions and compact inline controls. Font-size .82rem, radius 6px — intentionally smaller than full-size buttons.
Google OAuth — Special
<button class="google-btn">
  <!-- Google SVG icon -->
  Continue with Google
</button>
Fixed style per Google brand guidelines. Use only for OAuth sign-in/sign-up flows.

Info Blocks

Inline contextual messages. Left-border only — no background fill. Used inside forms to surface important notes.

Info (Blue)
The Content Manager account is not linked to your OLX account. 
<div class="info-block">
  <i class="bi bi-info-circle me-1"
    style="color:#1565c0;"></i>
  Your message here.
</div>
Success (Teal)
Password reset email sent to user@example.com.
Follow the link in the email to create a new password.
Didn't get it? Check your spam folder. 
<div class="info-block info-block-success">
  <i class="bi bi-check-circle me-1"></i>
  <strong>Bold first sentence.</strong><br>
  Second sentence.<br>
  Third sentence.
</div>
Structure: bold first line (the key fact), then plain supporting lines. Max 3 lines. Used on forgot-password and email-verify success states.
Toast Notifications

Appear at top-centre of the screen with a spring animation. Auto-dismiss after ~3 s. Use toast(message, type). Type: 'success' (default) or 'error'.

Live Demo
Code
// Success (default)
toast('Import completed successfully.');

// Error
toast('Something went wrong.', 'error');

// With language support
toast(currentLang === 'ua'
  ? 'Імпорт завершено'
  : 'Import complete');
Keep messages short — 1 sentence max. Toasts are not a replacement for inline validation errors on form fields.
